# CharacterArc 借鉴 novel-generator 的能力扩展总体规划

- 版本：`v0.1`
- 日期：`2026-05-05`
- 目标：在不放弃 CharacterArc 当前“本地优先 + 桌面创作工具 + 结构化小说工作台”定位的前提下，系统性吸收 `novel-generator` 在知识库、记忆、连续性校验、AI 可观察性、章节评审流水线、结构化学习、技能资产化方面的优势能力。
- 主要对标对象：`https://github.com/mushroomfk/novel-generator`

---

## 一、文档目的

本规划不是为了把 CharacterArc 改造成另一个 `novel-generator`，而是为了回答一个更实际的问题：

**哪些能力值得借，怎样借，借到什么程度最合适，如何分阶段落地而不把当前产品做散、做重、做乱。**

当前项目已经具备：

- 本地 SQLite 持久化
- 项目中心
- 世界观 / 角色 / 组织 / 关系 / 灵感 / 大纲 / 章节
- 章节工作台
- AI 助手与流式生成
- 写作风格与 project skills 基础
- 参考作品分析与“降低 AI 感”处理链

但和 `novel-generator` 相比，仍然缺少一层更厚的“AI 中间层”：

- 检索层
- 记忆层
- 连续性校验层
- 运行可观测层
- 分阶段评审层
- 结构化学习层
- 技能编排层

本规划的核心，就是补齐这 7 层。

---

## 二、总体判断

## 2.1 当前 CharacterArc 的优势

CharacterArc 当前更强的地方在于：

- 桌面端体验更完整
- 小说数据结构更直观
- 章节编辑器与 AI 助手工作流更贴近作者日常写作
- 中文小说 / 网文创作适配更强
- 本地单机体验更轻、更直接

它当前更像：

- **一个结构化、可落地、可实际写作的桌面小说创作工具**

## 2.2 novel-generator 值得借鉴的地方

`novel-generator` 真正强的，不是界面，而是中间能力：

- 本地知识库 / RAG
- 项目记忆自动沉淀
- 连续性 / 吃书 / OOC 校验
- AI 调用过程可追踪
- 章节评审流水线
- 导入资料后的结构化学习
- 技能模块化、资产化、可编排

它更像：

- **一个本地 AI 小说写作平台内核**

## 2.3 本次扩展的核心目标

这次不是“多加几个功能页”，而是把 CharacterArc 从：

- `结构化创作工具 + AI 调用入口`

升级为：

- `结构化创作工具 + 检索增强上下文 + 项目记忆 + 连续性审计 + 可编排 AI 创作流水线`

---

## 三、扩展原则

这次扩展必须严格遵守以下原则：

### 3.1 不改产品根定位

CharacterArc 仍然是：

- 本地优先
- 单机优先
- 作者工作台优先
- 小说流程优先

不改成：

- 平台型多用户系统
- 后端服务优先的 SaaS
- 资料管理过重的知识中台

### 3.2 不照搬 Tauri + FastAPI 双端结构

`novel-generator` 的很多能力来自它的后端服务层，但 CharacterArc 当前架构是：

- Electron main
- Vue renderer
- SQLite 本地持久化

本次扩展应优先保持当前架构一致性：

- **优先在 Electron main 内落地索引、检索、任务记录、分析编排**
- 不立刻拆出独立 Python 后端
- 除非出现明显算力、生态、维护性瓶颈，否则不引入第二套后端语言栈

### 3.3 流程优先，页面后置

新增能力先服务于当前流程：

- 参考资料 → 检索增强
- 章节创作 → 自动召回
- 生成结果 → 连续性校验
- 成稿推进 → 记忆沉淀
- 分析修稿 → 流水线执行

不要先扩成很多新页面，再倒逼流程去适配页面。

### 3.4 结构化优先，Prompt 后置

该做成结构化数据的，不要只做成 prompt 文本：

- 记忆要有类型
- 检索结果要有来源
- 校验结果要有问题类别
- 学习结果要有模板字段
- 技能要有输入输出声明

Prompt 仍然重要，但不能继续成为唯一的系统边界。

### 3.5 先可用，再自动化

每一层能力都先做：

1. 可手动触发
2. 可看见结果
3. 可验证有效
4. 再逐步自动化接入创作流程

避免一开始就做“全自动静默系统”，最后问题难排、用户也不信。

---

## 四、总能力地图

本次规划包含 7 条主线：

1. 本地知识库 / RAG 检索
2. 项目记忆层
3. 连续性 / 吃书校验
4. AI 过程可观察性
5. 章节评审流水线
6. 导入资料后的结构化学习
7. Prompt / 技能资产化

这 7 条主线不是平级乱做，而应形成上下游关系：

- **知识库 / RAG** 提供召回材料
- **项目记忆** 提供动态事实层
- **连续性校验** 负责生成后审计
- **AI 可观察性** 负责把过程记录下来
- **评审流水线** 把多个 AI 步骤串起来
- **结构化学习** 提供长期可复用的风格 / 模板资产
- **技能资产化** 把动作编排成稳定系统

---

## 五、能力一：本地知识库 / RAG 检索

## 5.1 目标

把当前“参考作品分析入口”升级为完整的项目级知识底座，使 AI 在生成前能够自动召回：

- 相关设定
- 前文章节
- 角色信息
- 组织关系
- 用户导入资料
- 参考作品片段
- 项目流程文档

从而减少：

- 忘设定
- 吃书
- 人设断裂
- 章节续写前文接不住

## 5.2 当前基础

当前项目已有基础：

- 参考作品分析入口
- 章节上下文拼装
- 世界观 / 角色 / 关系 / 大纲 / 章节等结构化数据
- Electron main 中的 AI transport / prompt 组装能力

但尚未具备：

- 通用资料导入索引
- 全文检索
- 向量检索
- AI 调用前的自动检索召回链

## 5.3 建议能力拆分

### A. 导入层

支持导入：

- txt
- md
- docx
- pdf
- html

后续可选：

- csv
- json
- epub

### B. 解析切块层

将导入资料切为 chunk，并保存：

- 原文片段
- 来源文件
- 资料类型
- chunk 序号
- 标题 / 章节名 / 段落标签
- 关键字
- 可选 embedding 向量

### C. 检索层

提供两类检索：

1. 全文检索（SQLite FTS）
2. 向量检索（embedding）

召回策略：

- keyword top-k
- vector top-k
- rerank
- 去重 / 合并

### D. AI 接入层

在 AI 任务执行前，根据任务类型自动召回：

- chapter-assistant：相关章节、角色、设定、资料片段
- chapter-first-draft：本章相关大纲、前文、人物状态、参考资料
- chapter-analysis：相关设定、前后章节、记忆事实
- workflow-documents：相关流程文件、参考资料、已生成资产

## 5.4 数据结构建议

建议新增表：

- `project_assets`
  - 项目资料元信息
- `project_asset_chunks`
  - 切块内容
- `project_asset_embeddings`
  - 向量数据（可选独立表）
- `retrieval_logs`
  - 某次 AI 调用召回了哪些内容

## 5.5 UI 入口建议

第一阶段不要急着做大型“知识库中心”，优先补 3 个入口：

1. 项目设置 / 参考资料导入区
2. 章节创作前的“已召回上下文”可视摘要
3. 搜索面板中增加“资料搜索”结果

## 5.6 验收标准

- 用户能导入 txt/md/docx/pdf/html
- 资料能被切块并存入本地索引
- AI 生成前能自动召回相关 chunk
- 用户能看见本次调用用了哪些资料
- 续写 / 分析场景下，AI 明显更少遗忘前文设定

---

## 六、能力二：项目记忆层

## 6.1 目标

让系统自动沉淀“动态事实”，而不是每次都只靠章节正文硬塞上下文。

记忆层要解决的问题：

- 某角色现在知道什么
- 某角色和另一人的关系发生了什么变化
- 本章到底发生了什么关键事件
- 某个组织立场是否已经公开
- 某条伏笔当前状态如何

## 6.2 当前问题

当前项目的数据很强，但偏静态：

- 世界观是静态设定
- 角色卡是静态介绍
- 关系表是静态结构
- 章节正文有动态信息，但不方便 AI 高效读取

缺少一层“从正文中抽出来的、可被复用的动态事实层”。

## 6.3 记忆类型建议

建议至少做这几类记忆：

1. **事件记忆**
   - 本章发生了什么
2. **认知记忆**
   - 谁知道了什么
3. **关系变化记忆**
   - 谁和谁更近 / 更远 / 结盟 / 结仇
4. **状态记忆**
   - 角色当前状态、伤势、位置、资源、任务
5. **悬而未决记忆**
   - 仍未解决的问题、风险、未回收伏笔

## 6.4 生成方式建议

第一阶段采取“半自动”：

- 章节保存版本时，可手动触发“沉淀记忆”
- 或在章节分析后，提供“生成本章记忆”按钮

第二阶段再做自动化：

- 本章定稿 / 保存版本后自动尝试抽取
- 用户确认后写入项目记忆

## 6.5 数据结构建议

建议新增表：

- `project_memories`
  - id
  - projectId
  - chapterId
  - type
  - title
  - content
  - subjects（角色/组织/地点等）
  - confidence
  - sourceSpan / sourceVersion
  - status（active / stale / superseded）

## 6.6 AI 接入方式

对于不同任务，记忆的接入方式不同：

- 续写：优先读最近章节记忆 + 当前角色状态
- 分析：读风险记忆 + 悬而未决记忆
- 关系冲突：读关系变化记忆 + 阵营立场记忆
- 章节初稿：读本卷进度记忆 + 关键剧情约束

## 6.7 验收标准

- 系统能把章节动态事实提炼成结构化记忆
- AI 在后续生成中能使用这些记忆
- 用户能看到、编辑、删除记忆
- 前后章节衔接明显提升

## 七、能力三：连续性 / 吃书校验

## 7.1 目标

在 AI 生成之后，不立刻把结果当成“可写回正文的正确答案”，而是增加一层自动审计。

审计重点：

- 人物 OOC
- 时间线断裂
- 战力跳变 / 资源跳变
- 信息越界
- 关系变化突兀
- 设定冲突
- 伏笔失管

## 7.2 当前基础

当前项目已经有：

- 章节分析 prompt
- 角色 / 关系 / 组织 / 大纲 / 世界观结构数据
- 章节上下文与相关章节信息

这说明校验层所需的大部分数据基础已经在，只是还没有形成“专项校验系统”。

## 7.3 校验输出格式

统一输出为：

- 问题
- 证据
- 影响
- 最小修法

必要时附：

- 风险等级
- 涉及角色 / 组织 / 章节
- 是否建议自动修正

## 7.4 校验模式建议

### 模式 A：手动校验

章节页新增动作：

- 连续性检查
- 吃书检查
- 人设检查

### 模式 B：生成后校验

在以下 AI 任务后提供一键审计：

- 本章初稿
- 续写片段
- 诊断改写
- 阵营视角

### 模式 C：定稿前总校验

对本章做一次综合校验：

- 角色状态
- 时间线
- 已知情报
- 关系变化
- 与上下章节是否冲突

## 7.5 数据结构建议

建议新增表：

- `continuity_reports`
- `continuity_report_items`

用于保存每次校验结果，避免结果只是临时消息。

## 7.6 验收标准

- 用户能对章节发起连续性审计
- 审计结果可复看、可定位、可执行
- 结果能引用具体章节内容、设定、记忆作为证据
- 定稿前可看到风险清单

---

## 八、能力四：AI 过程可观察性

## 8.1 目标

让用户和开发者都能知道：

- 本次 AI 到底读了哪些上下文
- 为什么输出这个结果
- 任务解析有没有失败
- 有没有跑修复 prompt
- 哪一步花时最多

这层能力对后续调 prompt、排查幻觉、优化体验都非常重要。

## 8.2 当前问题

当前项目更偏“只看结果”，过程相对黑盒：

- 上下文拼了什么，用户看不到
- 结构化解析失败，用户不一定知道
- 修复尝试发生过没有，不透明
- 检索命中是否靠谱，不透明

## 8.3 记录粒度建议

建议按一次 AI 调用保存 `ai_run` 记录，至少包含：

- task 类型
- quick action
- provider / model
- 输入摘要
- 上下文来源清单
- 检索命中列表
- prompt 阶段
- 输出摘要
- 解析 / 修复状态
- token / 耗时（若可得）
- 是否成功
- 错误信息

## 8.4 UI 呈现建议

第一阶段先做轻量视图：

- AI 回复卡片增加“查看过程”入口
- 展示：
  - 用到的章节
  - 用到的设定
  - 用到的资料
  - 运行阶段
  - 失败 / 修复记录

第二阶段再做：

- AI 调用历史面板
- 调试模式
- 过程过滤搜索

## 8.5 数据结构建议

建议新增表：

- `ai_runs`
- `ai_run_context_items`
- `ai_run_events`

## 8.6 验收标准

- 每次 AI 调用都可追溯
- 用户至少能看到“用了什么上下文”
- 开发者至少能看到“失败在哪一步”
- 修复与解析逻辑不再是纯黑盒

## 九、能力五：章节评审流水线

## 9.1 目标

把当前较分散的分析、润色、改稿动作，升级成有顺序、有边界、有结果物的章节评审流程。

建议标准流水线：

1. 初稿
2. 诊断
3. 连续性检查
4. 风格统一
5. 最终润色

## 9.2 为什么值得做

当前很多写作任务是“一次 prompt 赌结果”，存在问题：

- 产出不稳定
- 问题定位不清
- 修改理由不透明
- 用户难以选择只做其中一步

流水线的好处是：

- 每一步职责更清晰
- 用户可单独执行某一步
- 结果更容易留档、复用、追踪

## 9.3 建议产物

每一步最好都有对应产物：

- 初稿文本
- 诊断报告
- 连续性报告
- 风格修正建议
- 最终润色稿

不要所有结果都混在一个聊天消息里。

## 9.4 UI 入口建议

在章节工作台中增加“评审流程”入口，允许用户：

- 从某一步开始跑
- 单独重跑某一步
- 查看每一步结果
- 决定是否写回正文

## 9.5 数据结构建议

建议新增：

- `chapter_review_runs`
- `chapter_review_steps`

## 9.6 验收标准

- 用户能发起完整章节评审流程
- 每个步骤有独立结果
- 用户可跳步 / 重跑 / 选择性写回
- 最终流程比一次性大 prompt 更稳

---

## 十、能力六：导入资料后的结构化学习

## 10.1 目标

把当前“参考作品分析”从只生成风格总结，扩展成更系统的学习资产提炼器。

不仅提炼：

- 写作风格

还要提炼：

- 角色模板
- 节奏模板
- 场景模板
- 冲突模板
- 开篇钩子模板
- 章节结尾悬念模板
- 平台文风偏好

## 10.2 当前基础

当前已有：

- 参考作品分析入口
- 写作风格系统
- prompt library
- 项目级 skills

这意味着它已经有承接“结构化学习成果”的落点，只是分析产物还不够丰富。

## 10.3 学习结果建议分类

建议至少提炼成以下资产类型：

1. **Style DNA**
   - 语言特征、句式倾向、叙事镜头
2. **Pacing Pattern**
   - 节奏切换模式、爆点分布、铺垫/兑现比例
3. **Scene Pattern**
   - 环境描写方式、对话推进方式、转场手法
4. **Conflict Pattern**
   - 冲突来源、升级方式、反转方式
5. **Hook Pattern**
   - 开头钩子、章节结尾钩子、追读驱动方式

## 10.4 落地方式建议

学习结果不要只保存一段长文，而要结构化保存，并可挂接到：

- 项目默认写作风格
- 某个 prompt task 的附加约束
- 某个 project skill 的依赖资产

## 10.5 数据结构建议

建议新增：

- `reference_learning_assets`
- `reference_learning_asset_items`

## 10.6 验收标准

- 参考资料导入后可提炼多种模板资产
- 资产可在后续生成中被引用
- 用户能看见并选择启用哪些学习成果
- 写作风格系统不再只是一段 prompt 文本

---

## 十一、能力七：Prompt / 技能资产化

## 11.1 目标

把当前的 prompt 模板、快捷动作、project skills，从“文本片段集合”升级成“可声明、可依赖、可编排的任务资产”。

## 11.2 当前基础

当前已经有：

- promptLibrary
- chapterAssistantOptions
- project skills
- 项目级模板覆盖

方向是对的，但目前仍偏：

- 配置项
- prompt 文本
- 前端动作卡

还不是完整“技能定义系统”。

## 11.3 技能定义建议

每个 skill / task 至少声明：

- 名称
- 适用阶段
- 适用任务类型
- 输入要求
- 输出格式
- 依赖上下文
- 可选依赖资产
- 可选前置步骤
- 可选后置步骤

这样才能支持后续：

- 流程编排
- 结果校验
- 自动召回
- 多步骤流水线

## 11.4 目标形态

后续 AI 动作不应只是一条 prompt，而应是：

- 一个 skill definition
- 一个 runtime context builder
- 一个 output validator
- 一个 optional post-processor

## 11.5 数据结构建议

建议新增：

- `project_skill_definitions`
- `project_skill_runs`
- `project_skill_assets`

也可以先从 JSON 配置开始，再决定是否入库。

## 11.6 验收标准

- skill 不再只是文本模板
- 可以声明输入/输出/依赖
- 章节评审流水线可以复用这些 skill
- prompt 污染和模板失控问题明显下降

## 十二、分阶段实施顺序

这 7 条主线不能平均铺开，建议分 4 个阶段实施。

## 阶段 1：先补底座（必须最先做）

### 范围

1. 本地知识库 / RAG 检索
2. AI 过程可观察性

### 原因

如果没有检索层和运行记录层：

- 后面的记忆、审计、流水线都缺底座
- 出问题时也无法排查是哪一层错了

### 交付结果

- 资料导入与切块
- FTS 检索
- 可选 embedding 检索
- AI run 记录
- AI 上下文来源可视化

## 阶段 2：补动态事实层

### 范围

3. 项目记忆层
4. 连续性 / 吃书校验

### 原因

有了检索后，才有条件做更强的动态事实沉淀和校验。

### 交付结果

- 章节记忆提炼
- 关系/认知/状态记忆
- 连续性审计报告
- 定稿前风险清单

## 阶段 3：补创作流程层

### 范围

5. 章节评审流水线
6. 导入资料后的结构化学习

### 原因

前两阶段是基础能力，第三阶段开始把能力编排成对作者可感知的高价值工作流。

### 交付结果

- 初稿 → 诊断 → 校验 → 风格 → 润色 流程
- 参考作品学习资产化
- 结构化风格 / 节奏 / 冲突模板

## 阶段 4：补编排系统层

### 范围

7. Prompt / 技能资产化

### 原因

技能资产化要建立在前面几层已经稳定可用的基础上，否则只会把混乱 prompt 包装得更复杂。

### 交付结果

- skill definition
- 任务依赖声明
- 输入/输出约束
- 可复用 AI 流程编排能力

## 十三、推荐最小落地路线（MVP 顺序）

如果只按最小有效路径推进，建议按以下顺序做：

### MVP-1

- 资料导入
- 资料切块
- SQLite FTS 搜索
- AI 调用过程记录
- AI 上下文来源展示

### MVP-2

- 章节记忆提炼
- 章节连续性检查
- 风险报告保存

### MVP-3

- 章节评审流程（先做 3 步）
  - 初稿
  - 诊断
  - 连续性检查

### MVP-4

- 参考作品结构化学习
- 技能定义升级

## 十四、不建议现在做的事情

以下内容虽然未来可能有价值，但不建议在这轮规划里立刻做：

1. 多用户系统
2. 云同步
3. 在线协作
4. 平台化账号体系
5. 单独 Python 后端重构
6. 过重的资料树 / 资源中心
7. 一开始就全自动化所有流水线

原因很简单：

- 会把 CharacterArc 从桌面创作工具拉向平台产品
- 会显著增加维护成本
- 会打断当前已经基本可用的创作闭环

## 十六、最终结论

CharacterArc 最值得从 `novel-generator` 借鉴的，不是产品外壳，而是：

- **RAG 检索底座**
- **项目记忆层**
- **连续性审计层**
- **AI 可观察层**
- **章节评审流水线**
- **结构化学习资产层**
- **技能定义与编排层**

如果把这 7 层补起来，CharacterArc 会从当前的：

- `结构化小说创作桌面工具`

升级为：

- `具备检索增强、记忆沉淀、连续性校验、流程评审和技能编排能力的本地 AI 小说创作工作台`

这条路线比“盲目加更多 AI 按钮”更稳，也更能形成长期产品壁垒。
