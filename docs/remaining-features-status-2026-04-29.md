## 三、仍然要做的功能

## 3.1 P0：当前优先推进

### 1. 关系图可视化

当前状态：

- 关系组织的数据结构和 CRUD 已具备
- 搜索、导出、AI 上下文也已经能使用这些数据
- 但还没有图谱视图

剩余要做：

- 角色关系图谱视图
- 组织与角色连接可视化
- 节点筛选
- 关系强度视觉映射
- 点击图节点跳回资料卡

价值：

- 数据层已经成熟，现在做图不会返工
- 这是关系模块从“表单管理”走向“创作分析工具”的关键一步

### 2. 大纲优化

当前状态：

- 已有分卷管理
- 已有大纲节点 CRUD
- 已支持拖拽排序
- 已支持 AI 扩写单个大纲节点
- 已支持从大纲节点创建章节草稿

剩余要做：

- 更清晰的分卷到章节推进视图
- 更强的大纲节点状态与阶段管理
- 章节与大纲节点的联动回看
- 更完整的 AI 大纲工作流：
  - 按分卷生成节点
  - 基于当前章节回推后续节点
  - 批量补全节奏断点
- 更适合中长篇项目的结构检查能力

价值：

- 现在的大纲模块已经能用，但更偏“节点编辑器”
- 做完后会更像真正的创作规划中枢，而不只是剧情卡片列表

### 3. Prompt 模板优化

当前状态：

- 已有 prompt 代码层封装
- 已有写作风格系统
- 已有一批章节 AI 快捷动作
- 但还没有用户可配置的模板管理界面

剩余要做：

- 内置 prompt 模板列表
- 项目级 prompt 策略
- 快捷动作模板化
- 常用任务模板复用
- 模板的轻量编辑与切换能力

说明：

- 这不等于完整 Prompt 工坊
- 当前更合理的是先做本地轻量版优化

### 4. 伏笔 / 线索管理

当前状态：

- 当前还没有独立伏笔模块
- 代码中也未见对应数据结构、面板或导入导出支持

剩余要做：

- 伏笔条目数据结构
- 埋点章节 / 回收章节关联
- 状态管理：
  - 已埋下
  - 待回收
  - 已回收
- 与章节分析、关系组织联动

价值：

- 这是中长篇创作中非常实用的能力
- 也能和章节分析形成很强的互补

按当前代码状态看，CharacterArc 已经不再处于“基础面板未补齐”的阶段，而是进入：

- 核心创作闭环已完成

## 八、当前主线的可开发拆分

下面这部分用于把当前最优先的三条主线拆成可以直接进入开发的任务包。

### 8.1 关系图可视化

建议按两阶段推进：

#### 第一阶段：先做可用图谱 MVP

目标：

- 在现有关系组织模块上增加“图谱视图 / 列表视图”切换
- 不改数据库结构，直接复用现有：
  - organizations
  - characterRelationships
  - organizationMemberships
  - characters

建议任务：

- 新增关系图谱视图组件
- 将角色、组织统一转成节点
- 将角色关系、成员归属转成边
- 提供基础筛选：
  - 只看角色
  - 只看组织
  - 只看归属
  - 只看高强度关系
- 节点点击后联动右侧详情卡或回到原资料卡
- 关系强度先用线宽 / 透明度 / 颜色深浅表达

建议落点：

- `renderer/src/components/RelationsPanel.vue`
- 新增例如：
  - `renderer/src/components/RelationsGraphView.vue`
  - `renderer/src/features/relations/graph.ts`

第一阶段先不做：

- 自动布局高级算法优化
- 多层时间线
- 超大图谱性能优化

#### 第二阶段：补图谱可用性

目标：

- 让图谱不只是展示，而能服务创作分析

建议任务：

- 增加按角色 / 组织名搜索并聚焦节点
- 支持高亮某一角色的全部关系链
- 支持按组织聚焦成员网络
- 增加“冲突热点”视图
- 增加从图谱跳转到 AI 快捷动作：
  - 关系冲突
  - 阵营视角

### 8.2 大纲优化

建议按三步推进：

#### 第一步：补结构管理

目标：

- 把现有大纲模块从“节点列表”提升为“可追踪结构面板”

建议任务：

- 为大纲节点增加状态字段，例如：
  - idea
  - planned
  - drafting
  - done
- 增加章节绑定状态展示：
  - 未建章节
  - 已生成章节草稿
  - 已进入创作
- 在分卷下显示更明确的推进摘要

可能涉及：

- `renderer/src/types/app.ts`
- `renderer/src/stores/app.ts`
- `renderer/src/components/OutlinePanel.vue`
- 导入导出校验与标准化逻辑

#### 第二步：补联动能力

目标：

- 让大纲和章节之间形成双向回看

建议任务：

- 从章节反查对应大纲节点
- 在大纲节点上显示是否已有章节、章节状态、章节字数
- 在章节工作台中显示“来源大纲节点”
- 支持从章节更新回大纲摘要或推进状态

说明：

- 这一步会明显增强“规划 -> 写作 -> 回看”的连贯性

#### 第三步：补 AI 大纲工作流

目标：

- 不只生成单个节点，而是让 AI 能参与结构推进

建议任务：

- 按当前分卷批量补 3 到 5 个节点
- 基于当前章节生成下一段剧情链
- 基于已有节点做节奏断点检查
- 基于分卷目标生成缺失桥段建议

建议落点：

- `renderer/src/components/OutlinePanel.vue`
- `renderer/src/components/AiAssistantPanel.vue`
- `electron/main/aiPrompts.ts`

### 8.3 Prompt 模板优化

建议先做“轻量可配置版”，不要一开始做成复杂系统。

#### 第一步：抽象模板数据结构

目标：

- 让现在硬编码的快捷动作 prompt 可配置，但不破坏现有功能

建议任务：

- 新增 Prompt 模板类型定义
- 区分模板来源：
  - 内置模板
  - 项目级覆盖模板
- 支持模板字段：
  - id
  - label
  - group
  - prompt
  - mode
  - length
  - requiresSelection

可能涉及：

- `renderer/src/types/app.ts`
- `renderer/src/features/ai/chapterAssistantOptions.ts`
- `renderer/src/stores/app.ts`

#### 第二步：加轻量模板管理 UI

目标：

- 让用户能在项目内调整常用 AI 动作，而不是只能改代码

建议任务：

- 在设置页增加 Prompt 模板卡片区
- 支持：
  - 查看内置模板
  - 复制为项目模板
  - 编辑项目模板
  - 恢复默认模板
- 不做复杂排序系统，先保证可编辑和可恢复

建议落点：

- `renderer/src/components/SettingsPanel.vue`

#### 第三步：把模板接入提示词构建链路

目标：

- UI 层选择的模板真正进入 AI 请求，而不只是界面配置

建议任务：

- 章节快捷动作改为从模板表读取
- 让 `quickAction` 不只传 label，也能传 templateId
- 在提示词构建时优先使用项目模板文本
- 保留当前默认 prompt 作为兜底

关键落点：

- `renderer/src/components/AiAssistantPanel.vue`
- `renderer/src/features/ai/chapterAssistantOptions.ts`
- `electron/main/aiPrompts.ts`

### 8.4 推荐实际开工顺序

如果按“最小风险、最快见效”来排，建议不是三个方向并行平均做，而是：

1. 先做关系图可视化第一阶段
2. 再做 Prompt 模板第一步和第二步
3. 然后做大纲优化第一步
4. 最后再进入大纲 AI 工作流和 Prompt 深接入

原因：

- 关系图第一阶段基本不需要动数据库，见效最快
- Prompt 模板优化会碰 AI 流程，但可以先做轻量配置，不必一次重构到底
- 大纲优化最容易牵动类型、存储、章节联动和导入导出，适合放在前两项之后稳妥推进
